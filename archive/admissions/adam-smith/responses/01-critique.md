# Response to Problem 1: Critique

## Part 1: Objections to the Claim

**Measurement invalidity.** "Code shipped per week" is a proxy, and a particularly poor one for the phenomenon of interest. Code volume is easily increased by verbosity, scaffolding generation, and duplication-precisely the tasks where AI assistants are most fluent. If the 40% gain consists primarily of additional boilerplate, configuration files, and test stubs generated at AI prompting, the number may be real while representing near-zero additional productive value. To evaluate the claim seriously, we need to know: how was code measured (lines, commits, story points, PRs merged)? Was code quality assessed in parallel? What was the defect rate and review burden per unit of code in the treatment group versus the control?

**Selection bias on adopters.** Engineers who voluntarily adopt new tools differ systematically from those who do not. They are disproportionately technically confident, likely to work on standardized or boilerplate-heavy tasks, and self-selected for the kind of work where AI assistance provides most leverage. A study comparing willing adopters to non-adopters is measuring the combined effect of tool adoption and the antecedent characteristics that predict adoption. A credible causal estimate would require randomized assignment, a natural experiment, or at minimum a differences-in-differences design with pre-period data. The name of the specific evidence needed: baseline throughput data for both groups before tool adoption, and a plausible exclusion restriction.

**Task composition distortion.** AI coding assistants are unevenly effective across task types. They perform well at pattern completion, standard library usage, boilerplate generation, and familiar API calls. They perform poorly at novel algorithmic problems, debugging subtle system interactions, and reasoning about code whose context spans many files or which involves implicit domain constraints. A 40% throughput gain may therefore indicate not increased productivity on a fixed task distribution, but a shift in the task mix-more time on AI-compatible tasks, less on hard problems. This would mean the measured gain is partly a distortion artifact: the engineer's contribution to the difficult tasks that most need doing may have declined while the aggregate "code shipped" metric rose. What we would want: breakdown of throughput gains by task type, and some measure of how task mix shifted during the treatment period.

**Short time horizon and skill dynamics.** Even if the gain is real and unconfounded in the short term, the claim says nothing about the trajectory. There is a plausible mechanism by which AI assistance systematically reduces exposure to the parts of software engineering that are hardest and most skill-building: debugging unfamiliar codebases, constructing novel algorithms, reading and understanding poorly-documented systems. If that exposure is what produces expert software engineers over a career, then a tool that increases short-run throughput while reducing long-run skill development may produce a net negative. The evidence needed: longitudinal data on performance over 2–5 years among heavy AI assistant users compared to comparable engineers who did not adopt.

## Part 2: Even Granting the Claim, What Is Wrong with the Recommendation

The move from "engineers using AI assistants ship 40% more code on average" to "every engineer should adopt AI coding assistants" contains at least three logical errors.

First, an average does not license a universal recommendation. Averages aggregate a distribution. If the gain is concentrated in engineers working on boilerplate-heavy tasks, junior engineers, or specific tech stacks, then some engineers may gain substantially, others modestly, and others not at all or negatively. A recommendation to "every engineer" requires evidence about the distribution of effects, not just the mean. The recommendation form is wrong even if the number is right.

Second, the recommendation conflates individual and collective productivity. Code review, coordination, and system coherence are team-level properties. An engineer who ships 40% more code may impose review burden on colleagues disproportionate to the value of the additional code. If AI-generated code is harder to review, contains subtler defects, or is less consistent with existing system conventions, the gain to the individual may be a cost to the team. The correct unit of analysis for a staffing or tooling recommendation is not individual throughput but team output, which was not measured.

Third, and most simply: "more code" is not the goal, and the recommendation treats it as if it were. The goal is to solve problems, produce reliable systems, reduce future maintenance burden, and advance the organization's objectives. More code in service of those goals is good; more code that increases defect rates, degrades architecture, or fills review queues with AI-generated verbosity is not. A recommendation built on a proxy metric as if it were the terminal goal has confused the instrument for the thing being measured.
